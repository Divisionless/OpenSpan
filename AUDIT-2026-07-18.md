  reentrancy into Tk's wndproc that ended in the fail‑fast. Supported by: the stack, the
  two-builds-one-signature fact, and the correlation with "trying to move it" (the SendMessage drag).
  **[UNKNOWN]** the exact faulting call (no symbols).
- **Signature A (3:43)** = a *separate*, earlier `_ctypes`/`libffi` foreign-call access violation.
  Stack also shows `AudioSes.dll`. **[UNKNOWN]** whether this is the window at all — it could be the

## 5. The modern look — the evidence-based path (Doug wants it back)

The crash-prone parts were **window-STYLE manipulation** (`overrideredirect`, `SetWindowLong`+
`SWP_FRAMECHANGED`, `SendMessage` modal drag) that provokes the WM_ message storm into Tk. The
**safe** modern look uses **DWM attributes only** — single `DwmSetWindowAttribute` calls that never
touch window styles and never enter a modal loop:
- `DWMWA_USE_IMMERSIVE_DARK_MODE` (dark title bar) — already used by `_paint_dark_titlebar`, **never crashed.**
- `DWMWA_CAPTION_COLOR` / `DWMWA_TEXT_COLOR` — tint the caption to the app's palette.
- `DWMWA_WINDOW_CORNER_PREFERENCE` — rounded corners.

That yields a **dark, tinted, rounded native window** — modern, zero style-storm, zero crash risk.
A *fully* frameless window done safely is possible via `WM_NCCALCSIZE` subclassing (extend the client
over the caption, no style flip) — more work, a deliberate task, not a hunch.

**Recommendation:** ship the DWM-attribute dark caption now (safe, immediate) and, if the fully-frameless
look is wanted, do the `WM_NCCALCSIZE` version as its own scoped, tested piece — not mid-crisis.

## 6. Addendum 2026‑07‑19 — the crash was NOT the frameless (root cause found)

A crash on the **second Send-to-Tray** (dump `4216`, 2:27 PM) has the **identical signature** to §1's
signature B (`ucrtbase` 0xC0000409, offset 0x7286e) — but the frameless chrome had already been removed.
Stack walk: a **nested Tk event loop** (`tcl86t`/`_tkinter` twice, `user32` message dispatch) with
`_ctypes.pyd` at the base. **[PROVEN]**

Root cause, now identified: **reentrant Tk from a ctypes Win32 callback.** The just-added tray right-click
menu called `tk_popup` — which runs a *nested* Tk event loop — **directly inside the tray's ctypes
window-proc**, i.e. inside Windows message dispatch. Tk is not reentrant-safe there; it fail-fasts. The
code's own `self.ui` docstring says exactly this ("worker threads must NEVER touch Tk directly, not even
`after()`").

This **retro-explains §1**: the frameless drag used `SendMessage(WM_NCLBUTTONDOWN)`, which enters a modal
Windows message loop — the *same* reentrancy class. So the frameless was *a* source of this crash, not
*the* cause; removing it removed one source, and the tray menu was another. My §4 self-audit called the
crash `[INFERRED]` — correct to hedge, but I should have found this pattern (reentrant-Tk-from-ctypes)
rather than treating each instance as its own mystery.

**Fix shipped:** the tray menu is now deferred via `self.ui(...)` (the queue the Tk thread drains outside
any reentrant context); the tray window-proc now touches **no** Tk directly. **Confirmed by Doug — survives
all tray testing.**

## 7. Crystallized rules (the durable takeaways)

**R1 — Never call Tk from inside a ctypes Win32 callback or a modal message loop.**
A ctypes `WNDPROC`, or any path that enters a modal Windows loop (`SendMessage(WM_NCLBUTTONDOWN…)`,
`TrackPopupMenu`, `tk_popup` reached from such a context), runs *inside Windows message dispatch*. Tk is
not reentrant there and fail-fasts (`ucrtbase` 0xC0000409, offset 0x7286e). **Marshal ALL Tk work via
`self.ui(fn)`** — a plain queue the Tk thread drains every 50 ms outside any reentrant context. Even
`root.after()` from a worker thread is unsafe (GIL abort per the `self.ui` docstring); the queue is the
only safe marshal. This rule governs the pending #3 (`WM_NCCALCSIZE` is itself a window-proc subclass).

**R2 — Earn crash conclusions from the dump, never from a hunch.**
Full user-mode dumps land in `%LOCALAPPDATA%\CrashDumps\OpenSpan.exe.*.dmp`. Method: `Get-WinEvent`
(Application log) → faulting module + exception code; then walk the crashing thread's stack with the
Python `minidump` lib — tool checked in at `D:\OpenSpan\parse_dump.py`. A stack of
`tk86t`/`tcl86t`/`_tkinter` + `user32` + `_ctypes.pyd` ending in `ucrtbase` 0xC0000409 = the R1 bug.

**R3 — Cosmetic window chrome is the highest-risk / lowest-value work in this app; scope it, isolate it,
test it.** Four reworks for a "modern look" produced the ghost, the jitter, and the crash. The BLE
reconnect chain (bond-wipe → `KernelExperimental=false` → bounce → bond-persistence), diagnosed from real
logs at every step, held up. When touching the window again, do it alone, behind evidence, with the tray
as the escape hatch.

**Status:** #1 (honest per-indicator grey-out) ✅ · #2 (persistent tray + right-click controls) ✅
confirmed · #3 (fully-frameless via `WM_NCCALCSIZE`) pending, governed by R1.
